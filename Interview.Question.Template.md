# Interview.Question 템플릿 & 사용법

기존 `Interview.Question.md`의 질문·짧은답·긴답 형식을 유지하면서, 응답 품질을 일정하게 만드는 템플릿과 분류 목차를 제공합니다. 기존 문서는 그대로 두고, 본 템플릿으로 보완해 사용하세요.

---

## 🧰 답변 템플릿(STAR/CAR 혼합)
- 의도(Interviewer Intent): 질문이 노리는 역량/리스크.
- 한줄(Short): 8~14단어 핵심 메시지.
- 근거(Evidence): 수치·측정·재현 조건 2~3개.
- 구조(Approach): 선택(트레이드오프) → 구현(핵심 API/패턴) → 검증(지표/리그레션).
- 회고(Learnings): 다음엔 무엇을 바꿀지 1줄.

예시
```
의도: 체감 우선 설계를 했는지, 범위 관리 역량 확인.
한줄: “4일 내 ‘한 판 완주’를 위해 상태·이벤트·스킬을 표준화했습니다.”
근거: 33스크립트/약4천라인, OnChanged-*로 UI 자동 반영, 풀링+UniTask로 지연 최소화.
구조: 상태분리→이벤트동기→스킬확장, 취소 토큰으로 멈추면 즉시 정리.
회고: 다음엔 입력 버퍼/보정 로직을 확장해 피격 판정 신뢰도를 더 올리겠습니다.
```

## 🗂 분류 목차(권장 매핑)
- 설계/아키텍처(상태, 이벤트, 확장성, 의존성): Q1, Q3, Q9, Q10 …
- 성능/안정성(GC, 풀링, 애니, 취소): Q2, Q4, Q6 …
- 생산성/프로세스(우선순위, 시간관리, 검증): Q5, Q7, Q8 …
- 협업/유지보수(폴더/클래스 책임, 로깅): 관련 문항 전반.

> 팁: 각 질문 상단에 카테고리 태그 `[ARCH]`, `[PERF]`, `[PROC]`, `[COLL]`를 붙이면 복습 속도가 빨라집니다.

---

## 🧭 짧은답(30초) 만들기 규칙
- 문장 2~3개, “선택→근거→효과” 순서.
- 수치는 2개만: 기간/라인수/프레임·GC 중 상황 맞춤.
- 영어 로그/한글 주석 원칙을 언급하면(코드 기준) 통일성 어필에 유리.

서식
```
선택: 무엇을 우선했고 왜 그렇게 했는가.
근거: 실제 수치 1~2개(기간·규모·프레임·GC 등).
효과: 결합↓, 확장/검증 속도↑ 등 체감 결과.
```

---

## 🎛 라이브 데모 체크리스트(질문 대응용)
- 시연 순서 메모: 시작→스킬1~5(각 5~8초)→HP/피격→승패→리셋.
- 비상 단축키: `R`(리셋), `F1`(슬로우) 등.
- 관측 포인트: 상태 전환 로그, OnChanged-* 발생 로그, GC Alloc/FPS 표시.

---

## 🧩 예시 블록(질문 한 개에 붙여 쓰기)
```
의도: (예) 구조적 확장성 검증
한줄: BaseSkill로 신규 스킬 추가를 분 단위로 단축.
근거: 인터페이스·이벤트 표준화, 샘플 스킬 추가 시 변경 3파일 내.
구조: Prepare/Shot/End 이벤트 동기, UI는 OnChanged로 자동 반영.
회고: 피격 리액션 통합 규칙을 더 일찍 표준화했어야 함.
```
